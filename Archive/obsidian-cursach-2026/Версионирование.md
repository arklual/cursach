Раз ты хочешь хранить тяжелые конфиги (Python код, SQL, HTML шаблоны) именно **файлами на S3**, но при этом версионировать **"каждый чих"** (автосейвы/историю изменений), то тебе идеально подойдет паттерн **"Content-Addressable Storage" (CAS)** поверх S3.

Это тот же принцип, по которому работает Git внутри `.git/objects`, только вместо папки на диске у тебя бакет S3.

### Ключевая идея: S3 как хранилище уникальных блобов

Мы **не** создаем файл `workflow_v1.json`, `workflow_v2.json`. Это приведет к дублированию. Мы храним файлы по их **хэшу (SHA-256)**.

Если пользователь написал код `print("Hello")` — мы вычисляем хэш (например, `a1b2...`) и сохраняем файл с именем `a1b2...` на S3. Если в версии 100500 он снова написал `print("Hello")` — мы просто ссылаемся на **тот же самый файл** `a1b2...`.

---

### Архитектура

#### 1. S3 Бакет (Плоская структура)

В S3 хранятся только **неизменяемые** (immutable) файлы. Имя файла = Хэш содержимого.

Plaintext

```
s3://my-app-blobs/
  ├── 7f83b165... (файл с python кодом)
  ├── e3b0c442... (файл с sql запросом)
  └── 59d3e8a1... (файл с json конфигом)
```

#### 2. Postgres (Индекс и Скелет)

В базе мы храним **скелет** графа (JSON), но внутри этого JSON вместо тяжелого кода лежат ссылки на S3-ключи.

**Таблица `blob_index` (Кэш наличия файлов)** Чтобы на каждый "чих" не долбить S3 проверкой `headObject` (существует ли файл?), мы храним индекс в базе.

SQL

```
CREATE TABLE blob_index (
    content_hash CHAR(64) PRIMARY KEY, -- SHA-256
    s3_key VARCHAR(255) NOT NULL,
    created_at TIMESTAMP DEFAULT NOW()
);
```

**Таблица `workflow_revision` (История каждого чиха)**

SQL

```
CREATE TABLE workflow_revision (
    id BIGSERIAL PRIMARY KEY,
    workflow_id UUID REFERENCES workflow(id),
    revision_number INT, -- 1, 2, 3... (автоинкремент для UI)
    
    -- Легковесный JSON.
    -- Внутри: координаты узлов, связи и S3-ХЭШИ конфигов.
    graph_skeleton JSONB NOT NULL, 
    
    created_at TIMESTAMP DEFAULT NOW()
);
```

### Алгоритм работы (The Flow)

#### Сценарий 1: Пользователь редактирует код (Autosave)

Пользователь печатает код в Python-ноде. Срабатывает дебаунс (автосейв).

1. **Frontend:** Шлет на бэк полный JSON графа + контент измененной ноды.
    
2. **Backend:**
    
    - Вычисляет `HASH = SHA256(node.content)`.
        
    - **Проверка:** `SELECT 1 FROM blob_index WHERE content_hash = :HASH`.
        
    - **Если нет в базе:**
        
        - Загружает контент в S3 -> `PUT /bucket/HASH`.
            
        - Пишет в `blob_index` -> `INSERT VALUES (HASH)`.
            
    - **Если есть:** Ничего не делает с S3 (экономим трафик и деньги).
        
3. **Backend (Сохранение ревизии):**
    
    - Формирует `graph_skeleton`: заменяет тело кода на `HASH`.
        
    - `INSERT INTO workflow_revision ...`.
        

**Итог:** "Чих" стоит дешево. Мы пишем только легкую строку в Postgres. В S3 пишем только если код реально изменился и стал уникальным.

#### Сценарий 2: Получение диффа (Diff)

Пользователь хочет узнать: "Что изменилось между версией 50 и 55?".

1. **Backend:**
    
    - Берет `skeleton_50` и `skeleton_55` из Postgres.
        
    - Сравнивает JSON-структуры.
        
2. **Логика сравнения:**
    
    - Если координаты изменились — помечаем как `visual_change`.
        
    - Если `config_hash` у ноды отличается (`abc...` vs `xyz...`) — это `content_change`.
        
3. **Детализация:**
    
    - Фронт видит, что хэши разные.
        
    - Фронт делает запрос за контентом (или генерирует Presigned Link на S3) для двух файлов: `s3://.../abc` и `s3://.../xyz`.
        
    - Фронт (или бэк) делает текстовый diff двух строк.

### Почему это круто для твоего кейса

1. **S3 стоит копейки:** Ты не плодишь дубликаты. Если у тебя 1000 версий воркфлоу, и в 999 из них SQL-запрос не менялся — на S3 лежит **один** файл.
    
2. **Мгновенные дифы:** Тебе не нужно качать файлы с S3, чтобы понять, _были ли_ изменения. Ты просто сравниваешь строки хэшей в базе (O(1)). Качать надо только для отображения конкретных строк кода.
    
3. **Отслеживание "каждого чиха":** Вставка JSON-скелета (без кода) в Postgres очень быстрая. Ты можешь сохранять хоть каждую секунду.
    

### Что делать с "мусором" в S3?

Так как мы пишем каждый уникальный чих, S3 будет расти. **Решение:** Настрой **S3 Lifecycle Policy**.

- Раз в сутки запускай джобу "Cleanup":
    
    1. Проверь, какие хэши используются в `workflow_deployment` (активные версии) и последних N ревизиях.
        
    2. Все файлы, на которые нет ссылок в базе и которые старше 30 дней — удаляй.
        
- Или просто переводи старые объекты в **S3 Glacier** (холодное хранилище) через 30 дней — это супер дешево.
    

Начать внедрение лучше с создания таблицы `blob_index` и метода `uploadIfNotExist(content)`.
